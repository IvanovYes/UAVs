# UAVs

Макет ПО управлению группой БВС.
Архитектура ПО показана в следующей структурной схеме:

<img width="811" height="158" alt="image" src="https://github.com/user-attachments/assets/1ecf6776-f48b-4078-acb5-fd834fcae0a5" />


ПО состоит из двух незаисимых исполняемых частей:

Сервера - центрального узла, который управляет группой БВС.

Клиентов - отдельных экземпляров ПО, запущенных на каждом БВС (на Raspberry Pi или аналогичном бортовом компьютере), и соединяемых с полетным контроллером.

Из двух (по моему мнению) возможных вариантов:

Классическая архитектура (сервер ожидает (не зная ничего об адресах и портах клиентов), клиенты подключается). При этом для работы в режиме реального времени использутся не классическая запрос-ответная система, а непрервная многопоточная (как минимум поток на прием и сохранение в буфер, другой на обработку и ответ).

Гибридная архитектура (сервер и клиенты взаимно инициируют соединение друг к другу через последовательный вызов команд bind и connect, то есть сервер заранее знает об адресах и портах всех клиентов и создает не один UDP-сокет, а целый массив, по одному на каждого клиента, так называемые "связанные UDP-сокеты". При этом для приема данных сединение на каждом сокете из массива запускается в отдельном программном потоке, либо в одном потоке используется poll() или select() на массиве этих сокетов (которые будут работать с микросекундными задержками), что в любом случае автоматически обеспечивает прием данных только от требуемых источников клиентов-БВС, потому что связь с ними устанавливается заранее.

Была выбрана классическая архитектура, так как БВС моежт иметь динамический IP. 

Таким образом, понадобится проверка входящих пакетов данных на вхождение в список БВС, формируемый до начала сессии.
Сервер будет работать в трехпоточном режиме:

<img width="1247" height="200" alt="image" src="https://github.com/user-attachments/assets/abbd3d77-7168-4cfb-b489-645cf81c1836" />


Главный поток - непрерывно выводит информацию об состоянии группы БВС, а также принимает пользовательский ввод.
принимающий поток - непрерывно принимает данные с клиентов и записывает их в буфер - встроенную (реализованую непосредственно в коде) - динамически обновляемую БД задаваемых клиентах-БВС до начало выполнения полетного задания).

Управляющий поток - непрерывно обрабатывает информацию из буфера и принимает входящие запросы от пользователя, на основании чего шлет команды всей группе либо отдельному клиенту БВС.  

Управляющая поток сервера реализован в виде автомата конечных состояний:

Ввод:
- клиентов
- полетного задания

Команды:
- проверка соединения и верификация каждого БВС группы
- начать миссию
- закончить миссию
- вернуться на базу
- остановить миссию

Состояния:
- группа не готова
- группа готова
- группа взлетает
- группа садиться
- прерывания действий и зависание группы
- выполнение миссии группой
- возврат на базу группы
- авария
